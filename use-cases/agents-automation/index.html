Use cases / Automation

Scripts and agents do not read your terminal theme

Automation needs stable contracts: which port, which database name, whether the process is listening, and when it is safe to hit HTTP. Treeline exposes that as structured data and socket-controlled lifecycle—not string parsing of gtl status output.

Interfaces that scale

  • MCP — gtl mcp exposes start, stop, restart, port, status, and related tools to editors. Prefer this when the runtime speaks MCP.
  • JSON flags — gtl status --json, gtl doctor --json, gtl port --json, gtl db name --json, gtl env --json, gtl resolve <project> --json for anything that runs shell only.
  • Readiness — gtl start --await blocks until TCP accepts on the allocated port (with --await-timeout). Use after start in CI or agent loops.
  • Env inspection — gtl env prints the worktree env file; --template shows unresolved tokens.
  • Completions — gtl completion bash|zsh|fish|powershell; Homebrew installs completions on supported platforms.

gtl init writes an AGENTS.md file with instructions for setup, teardown, and server management. Any MCP-compatible editor reads it as context. Cross-repo service URLs for agents follow the same gtl resolve contract as humans.
