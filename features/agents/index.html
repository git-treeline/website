AI Agents

Your agent gets the
same tools you do.

A built-in MCP server gives agents native access to worktree management. Lifecycle hooks, structured output, and auto-generated context files make every agent environment-aware.

Agents don't know about your environment

Your AI agent creates a worktree and tries to start the server. It doesn't know about port allocation, database cloning, or env files.

agent without treeline

agent $ git worktree add ../fix-bug fix-bug

agent $ cd ../fix-bug && npm start

Error: EADDRINUSE :::3000

# Agent tries a random port...

agent $ PORT=4567 npm start

Warning: DATABASE_URL not set

# Agent guesses at database names...

agent $ createdb fix_bug_dev

# Works sometimes. Breaks often.

Three lines in your editor config

MCP config (Cursor / Claude Code / Codex) 
{
  "mcpServers": {
    "gtl": {
      "command": "gtl",
      "args": ["mcp"]
    }
  }
}

Now the agent calls start, port, status as native tools under the gtl server namespace. No shell parsing, no guessing, no string manipulation.

What this makes possible

Cursor restarts your server

Agent edits a file, calls restart, confirms the server is healthy via status. No shell, no PID hunting.

Full lifecycle automation

Claude Code creates a worktree with gtl setup ., modifies code, runs tests, tears down with gtl release . --drop-db. Clean sandbox every time.

Multi-agent orchestration

An orchestrator spins up five agents in five worktrees. Each has isolated ports, databases, and a controllable server. All managed via structured API.

Four integration surfaces

Different agents need different interfaces. Treeline supports all of them.

MCP server

gtl mcp starts a Model Context Protocol server that exposes worktree tools and resources. Agents call start, stop, restart, port, status as native tool calls — no shell required.

Resources expose gtl://allocations and gtl://config/user for read access to the full registry and user-level configuration.

start
Boot the server via the supervisor
stop
Stop the running server
restart
Restart the server
status
Allocation details and supervisor state
port
Get the allocated port for this worktree
list
All worktrees and their allocations
doctor
Diagnose configuration issues
db_name
Get the allocated database name
config_get
Read a config key from .treeline.yml

Lifecycle hooks

gtl setup . allocates resources and runs setup commands for the current directory — the worktree creation hook. gtl release . --drop-db is the teardown hook that drops the database and cleans up.

Agents that create and destroy worktrees call these at the boundaries. Everything in between is their own code.

agent lifecycle

# Create + setup

$ git worktree add ../fix fix-branch

$ cd ../fix && gtl setup .

Resources allocated. Ready.

# ... agent does its work ...

# Teardown

$ gtl release . --drop-db

Released. Database dropped.

Structured output

gtl status --json returns the full allocation registry as JSON. For richer structured access — ports, database names, config — agents use the MCP tools and resources directly.

$ gtl status --json

[{

  "branch": "feature-auth",

  "port": 3010,

  "database": "myapp_dev_feature_auth",

  "project": "myapp",

  "worktree_name": "feature-auth"

}]

Agent context files

gtl init generates an AGENTS.md file that tells Cursor, Claude Code, and Codex how to use Treeline. The agent reads it on startup and knows exactly which commands to run for setup, teardown, and server management.

$ gtl init

Created .treeline.yml

Created AGENTS.md

# AGENTS.md includes:

# - Setup: gtl setup .

# - Teardown: gtl release . --drop-db

# - Port: gtl port

# - Server: gtl start / stop / restart

Connect your agent

Add the MCP config to your editor. The agent discovers everything else automatically.
