Networking

Your OAuth redirect URI works
on every branch.

Integrations expect stable URLs. Worktrees have ephemeral ports. Three features bridge the gap at different layers.

Every integration breaks when you change ports

You register localhost:3000 as your OAuth callback. You switch to a worktree on port 3010. Login breaks.

feature-auth on :3010

OAuth callback received at localhost:3000

ERR: No server listening on port 3000

# Redirect URI was registered as localhost:3000.

# This worktree runs on :3010.

# Change the URI? Break the other worktree.

Stripe webhooks, Mapbox keys, CI callbacks, SSO redirects — all hardcoded to a port or hostname that keeps changing between worktrees.

The URL follows the branch, not the port

localhost:3010

feature-auth.localhost

Stable local name

feature-auth.example.dev

Public URL

Each branch gets a stable URL like https://feature-auth.localhost/callback. Register specific redirect URIs per branch, or point them at a fixed port with gtl proxy. No more port-swapping between worktrees.

What this makes possible

Stable URLs per branch

Each branch gets a named URL like feature-auth.localhost. Register the specific redirect URI, or use gtl proxy to forward a fixed port to whichever worktree you choose.

Share a running branch

Send feature-auth.example.dev to a teammate. They see the running app from your worktree, no deploy required. QA tests from their phone.

External services reach you

Cloudflare tunnel gives your local worktree a public URL. Payment processors, CI callbacks, mobile app deep links — all routed to your branch.

Same problem, three layers

serve and tunnel route automatically. proxy is a manual escape hatch for port-pinned services. Use one, two, or all three.

gtl serve

Local HTTPS subdomain router

A background router that maps {branch}.localhost to the right worktree port. HTTPS with a trusted local CA. Runs as a system service — install once, forget about it.

Best for: OAuth redirects, API keys tied to a hostname, day-to-day development where you want named URLs instead of port numbers.

$ gtl serve install

Generated trusted CA certificate

Installed system service

Listening on https://*.localhost

gtl proxy

Manual port forwarder

Forwards all traffic from a fixed port (e.g. localhost:3000) to a worktree's allocated port. You choose the target — run it from a worktree directory and it infers the port, or specify both explicitly. Optional TLS via mkcert.

Best for: External services hardcoded to localhost:3000 that can't use subdomains — OAuth providers, webhook endpoints, API keys tied to a port number.

$ gtl proxy 3000

Proxying :3000 → :3050

gtl tunnel

Public URLs via Cloudflare

Makes your local worktree publicly accessible at {branch}.example.dev. Uses Cloudflare tunnels under the hood — edge TLS, no port forwarding, no ngrok.

Best for: Sharing a running branch with a teammate, mobile testing, external service integrations that need a public URL.

$ gtl tunnel

Tunnel active

https://feature-auth.example.dev

Most teams only need gtl serve. Install once, every branch gets a name.
