git-treeline

Review every branch at the same time

Agents build features in worktrees faster than you can review them. Git Treeline allocates server ports, clones databases, isolates background job processes, and more so your team can run every branch, across every repo, with zero collisions.

Worktrees are easy to create.
Hard to run.

git worktree add gives you a second checkout in seconds. But that checkout isn't runnable. It needs its own port, its own database, its own env file, its own dependencies installed. Until all of that is in place, nobody can open it in a browser and see what was built.

Three feature branches ready for review, zero of them running. That's the gap Git Treeline closes.

Everything a worktree needs to run

Each capability is opt-in. Use what your project needs, skip what it doesn't.

Port allocation

Assigns a unique port per worktree from a configurable base. Multi-port support for apps that need it — like Rails with a separate esbuild port.

PORT=3010 ESBUILD_PORT=3011

Database cloning

Clones your development database per worktree. PostgreSQL via createdb --template, SQLite via file copy. Full schema and seed data, zero migration overhead.

myapp_dev_feature_auth

Redis namespacing

Prevents key collisions across worktrees. Choose key-prefix strategy (unlimited worktrees) or separate Redis databases (1–15).

myapp:feature-auth:session:*

Env file management

Copies your env template, interpolates allocated values (ports, database names, Redis URLs), and writes the result. Token-based — works with any .env format.

.env.local ← .env.local + allocations

Setup commands

Runs your install and migration commands after allocation. bundle install, npm install, prisma migrate deploy — whatever your stack needs.

setup_commands: [...]

File copying

Copies files from the main worktree that aren't in git — secrets, credentials, local config. No more "where's my master.key" after checkout.

copy_files: [config/master.key]

From checkout to running app

Create worktrees, review pull requests, monitor environments, and clean up — all from the CLI. Every command designed for both human developers and automated agents.

git-treeline installs a short alias — gtl — so you'll see it used from here on.

create + setup in one step

$ gtl new feature-auth --start

release

Free ports, drop the cloned database, release Redis namespace. Batch with --project or --all.

prune

Remove allocations for deleted worktrees. Use --merged to target branches already merged to main.

status --watch

Live-updating dashboard with port health checks. Set refresh rate with --interval.

Declare what your project needs

Commit .treeline.yml to your repo. Every team member and every agent gets the same setup.

gtl init auto-detects your framework — Next.js, Rails, Express, Python, Rust, Go — and generates a tailored config.

.treeline.yml — Rails app 
project: myapp
merge_target: develop  # auto-detected; set if not main
ports_needed: 2

env_file:
  target: .env.local
  source: .env.local

database:
  adapter: postgresql
  template: myapp_development
  pattern: "{template}_{worktree}"

copy_files:
  - config/master.key

env:
  PORT: "{port}"
  DATABASE_NAME: "{database}"
  REDIS_URL: "{redis_url}"
  ESBUILD_PORT: "{port_2}"

setup_commands:
  - bundle install --quiet
  - yarn install --silent

start_command: bin/dev
.treeline.yml — Next.js + Prisma 
project: frontend

env_file:
  target: .env.local
  source: .env.local

database:
  adapter: postgresql
  template: frontend_dev
  pattern: "{template}_{worktree}"

env:
  PORT: "{port}"
  DATABASE_URL: "postgresql://localhost/{database}"

setup_commands:
  - npm install
  - npx prisma migrate deploy

start_command: npm run dev

Framework agnostic

Rails reads dotenv-rails, Next.js reads .env.local, Express reads .env. Git Treeline writes the file your stack already loads. No extra dependencies.

Questions

What does git-treeline do?

Git Treeline is a CLI tool that sets up isolated development environments per git worktree. It allocates unique ports, clones PostgreSQL or SQLite databases, assigns Redis namespaces, writes env files with interpolated values, copies secret files, and runs setup commands. One command turns a bare worktree checkout into a bootable copy of your app.

Do I need to use all the features?

No. Every capability is opt-in. If your app only needs port allocation and an env file, that's a three-line .treeline.yml. Database cloning, Redis namespacing, file copying, and setup commands are all configured only if you need them.

What's the difference between setup and new?

gtl new creates a git worktree and sets it up in one step — it runs git worktree add followed by full resource allocation and setup. gtl setup only allocates resources for an existing worktree. Use new when starting fresh, setup when you've already created the worktree yourself.

Can I use git-treeline with SQLite?

Yes. Set database.adapter: sqlite in your .treeline.yml. Treeline clones SQLite databases by copying the file, and cleans up on release. PostgreSQL uses createdb --template for the same effect. Both adapters give each worktree a fully isolated database.

How do I review a GitHub PR with git-treeline?

Run gtl review <PR-number>. Treeline uses the gh CLI to fetch the PR, creates a worktree for it, allocates isolated resources, and runs setup. Add --start to boot the app immediately. When you're done, gtl release --drop-db cleans everything up.

How does database cloning work?

For PostgreSQL, Git Treeline uses createdb --template to clone your development database per worktree. This copies the full schema and seed data without running migrations, so each worktree gets a complete database in seconds. For SQLite, it copies the database file. Use gtl release --drop-db to clean up when you're done.

Does it work with AI coding agents?

Yes. Git Treeline is designed for it. Add gtl setup . as a lifecycle hook when an agent creates a worktree, and gtl release . --drop-db on teardown. Use --json output for programmatic access to allocated resources. gtl init can also generate agent context files for Cursor and Claude so agents understand the Treeline setup.

Does git-treeline work with Cursor and Claude Code?

Yes. When you run gtl init, it can generate agent context files — .cursor/rules/treeline.mdc for Cursor or a CLAUDE.md section for Claude Code. These files teach AI agents how to use Treeline commands in your project. Use --skip-agent-config to opt out.

What happens if I forget to clean up?

gtl prune --stale scans for allocations whose worktree directories no longer exist and removes them. gtl prune --merged goes further and releases allocations for branches that have been merged to main. It's garbage collection for your dev environment.

How do I clean up merged branch worktrees?

Run gtl prune --merged to detect and release allocations for worktrees on branches that have been merged to main. Add --drop-db to also remove their cloned databases. For bulk cleanup, gtl release --all --drop-db releases everything across all projects.
