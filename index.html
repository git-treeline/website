Everything a worktree needs to run

Each capability is opt-in. Declare what your project needs in .treeline.yml, skip what it doesn't.

Port allocation

Assigns a unique port per worktree from a configurable base. Multi-port support for apps with separate asset build processes.

PORT=3010 ESBUILD_PORT=3011

Database cloning

Clones your development database per worktree. PostgreSQL via createdb --template, SQLite via file copy. Reset, restore from a dump, or clone from a different source with gtl db.

gtl db reset · gtl db restore dump.sql

Environment isolation

Writes your .env file with interpolated ports, database names, and Redis URLs. Copies secrets from the main worktree. Runs setup commands with all allocated values available.

.env.local ← allocated values

Human-visible, agent-controllable

gtl start runs your dev server under a supervisor with allocated env vars injected directly into the process — no .env file reading required. Agents can stop and restart it remotely while you watch the logs in your own terminal. No hidden processes, no detached sessions.

Every worktree gets a URL

Stop remembering port numbers. Git Treeline maps branches to named URLs — locally with HTTPS, or publicly through Cloudflare tunnels.

Local HTTPS router

gtl serve maps every worktree to {branch}.localhost with trusted certificates. Always-on background service, zero config per branch.

https://feature-auth.localhost

Port forwarding

gtl proxy binds a fixed port and forwards to the current worktree. OAuth callbacks, webhooks, and Stripe all point to :3000 regardless of which branch you're running.

gtl proxy 3000 --tls

Public tunnels

gtl tunnel exposes a running branch at a public URL via Cloudflare. Branch-named subdomains on your own domain, or a quick random URL with no account.

https://feature-auth.myteam.dev

Zero footprint until you need it

Git Treeline is invisible until you need it, and out of the way once it's done.

No runtime dependency

Git Treeline is a CLI tool, not a library. It doesn't ship in your bundle, your Docker image, or your production deploy. It writes a .env file and walks away.

No lock-in

.treeline.yml is declarative config that sits in your repo. Developers who don't have gtl installed just ignore it. The app works exactly as before.

No magic

No git hooks. No background daemons. No shell integrations. No implicit behavior. You run gtl setup, it allocates resources and writes your env file. That's it.

Opt-in per developer

Commit the config, let the team decide. Developers who work in worktrees install gtl and get isolated environments. Everyone else doesn't notice it exists.

Built for AI agents

Two commands plug Git Treeline into any agent orchestrator. Allocate on worktree creation, tear down on completion. Structured output and auto-generated context files keep agents informed.

Lifecycle hooks

gtl setup and gtl release --drop-db map directly to agent worktree creation and teardown.

Machine output

--json on any read command. Orchestrators get structured data — ports, databases, paths — without parsing.

Agent context

gtl init writes an AGENTS.md section so Cursor, Claude Code, and Codex agents know to use gtl port for port discovery.

Questions

What does git-treeline do?

Git Treeline is a CLI tool that sets up isolated development environments per git worktree. It allocates unique ports, clones PostgreSQL or SQLite databases, assigns Redis namespaces, writes env files with interpolated values, copies secret files, and runs setup commands. One command turns a bare worktree checkout into a bootable copy of your app.

Do I need to use all the features?

No. Every capability is opt-in. If your app only needs port allocation and an env file, that's a three-line .treeline.yml. Database cloning, Redis namespacing, file copying, and setup commands are all configured only if you need them.

How do I review a GitHub PR with git-treeline?

Run gtl review <PR-number>. Treeline uses the gh CLI to fetch the PR, creates a worktree for it, allocates isolated resources, and runs setup. Add --start to boot the app immediately. When you're done, gtl release --drop-db cleans everything up.

Does it work with AI coding agents?

Yes. Git Treeline is designed for it. Add gtl setup . as a lifecycle hook when an agent creates a worktree, and gtl release . --drop-db on teardown. Use --json output for programmatic access. gtl init generates agent context files for Cursor and Claude so agents understand the environment.

Does git-treeline add runtime dependencies to my app?

No. Git Treeline is a standalone CLI binary. It doesn't ship in your bundle, your Docker image, or your production deploy. It writes a .env file at setup time and nothing else. Your app has zero awareness that Treeline exists.

What if some team members don't use git-treeline?

Nothing changes for them. .treeline.yml is a config file in your repo that they can ignore. The app still boots on default ports with the default database. Developers who work in worktrees install gtl and get isolated environments. Everyone else keeps their existing workflow.
