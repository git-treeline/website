Worktrees are easy to create. Hard to run.

git worktree add gives you a second checkout in seconds. But that checkout isn't runnable. It needs its own port, its own database, its own env file, its own dependencies installed. Do that by hand for three worktrees and you've burned an hour before writing any code.

Now multiply that by AI agents. Three agents building features in parallel, each in their own worktree — and you can't review their work until each one boots without colliding.

Everything a worktree needs to run

Each capability is opt-in. Use what your project needs, skip what it doesn't.

Port allocation

Assigns a unique port per worktree from a configurable base. Multi-port support for apps that need it — like Rails with a separate esbuild port.

PORT=3010 ESBUILD_PORT=3011

Database cloning

Clones your development database per worktree using createdb --template. Full schema and seed data, zero migration overhead.

myapp_dev_feature_auth

Redis namespacing

Prevents key collisions across worktrees. Choose key-prefix strategy (unlimited worktrees) or separate Redis databases (1–15).

myapp:feature-auth:session:*

Env file management

Copies your env template, interpolates allocated values (ports, database names, Redis URLs), and writes the result. Token-based — works with any .env format.

.env.local ← .env.local + allocations

Setup commands

Runs your install and migration commands after allocation. bundle install, npm install, prisma migrate deploy — whatever your stack needs.

setup_commands: [...]

File copying

Copies files from the main worktree that aren't in git — secrets, credentials, local config. No more "where's my master.key" after checkout.

copy_files: [config/master.key]

Declare what your project needs

Commit .treeline.yml to your repo. Every team member and every agent gets the same setup.

.treeline.yml — Rails app 
project: myapp
ports_needed: 2

env_file:
  target: .env.local
  source: .env.local

database:
  adapter: postgresql
  template: myapp_development
  pattern: "{template}_{worktree}"

copy_files:
  - config/master.key

env:
  PORT: "{port}"
  DATABASE_NAME: "{database}"
  REDIS_URL: "{redis_url}"
  ESBUILD_PORT: "{port_2}"

setup_commands:
  - bundle install --quiet
  - yarn install --silent
.treeline.yml — Next.js + Prisma 
project: frontend

env_file:
  target: .env.local
  source: .env.local

database:
  adapter: postgresql
  template: frontend_dev
  pattern: "{template}_{worktree}"

env:
  PORT: "{port}"
  DATABASE_URL: "postgresql://localhost/{database}"

setup_commands:
  - npm install
  - npx prisma migrate deploy

Full lifecycle, not just setup

Worktrees accumulate. Databases pile up. Ports stay reserved. Git Treeline tracks everything in a central registry and gives you commands to manage the full lifecycle.

terminal

~ $ git-treeline status

┌──────────────────────┬──────┬──────────────────────────┬───────────────────────┐

│ Worktree │ Port │ Database │ Redis │

├──────────────────────┼──────┼──────────────────────────┼───────────────────────┤

│ myapp (main) │ 3000 │ myapp_dev │ default │

│ myapp-feature-auth │ 3010 │ myapp_dev_feature_auth │ myapp:feature-auth │

│ myapp-fix-payments │ 3020 │ myapp_dev_fix_payments │ myapp:fix-payments │

└──────────────────────┴──────┴──────────────────────────┴───────────────────────┘

~ $ git-treeline release ../myapp-fix-payments --drop-db

Released port 3020

Dropped database myapp_dev_fix_payments

Released Redis namespace myapp:fix-payments

~ $ git-treeline prune --stale

Pruned 2 allocations for deleted worktrees

setup

Allocate resources, write env files, run install commands. Idempotent — safe to re-run.

release --drop-db

Free ports, drop the cloned database, release Redis namespace. Clean teardown.

prune --stale

Find and remove allocations for worktrees that no longer exist. Garbage collection for your dev environment.

Questions

What does git-treeline do?

Git Treeline is a CLI tool that sets up isolated development environments per git worktree. It allocates unique ports, clones PostgreSQL databases, assigns Redis namespaces, writes env files with interpolated values, copies secret files, and runs setup commands. One command turns a bare worktree checkout into a bootable copy of your app.

Do I need to use all the features?

No. Every capability is opt-in. If your app only needs port allocation and an env file, that's a three-line .treeline.yml. Database cloning, Redis namespacing, file copying, and setup commands are all configured only if you need them.

How does database cloning work?

Git Treeline uses PostgreSQL's createdb --template to clone your development database per worktree. This copies the full schema and seed data without running migrations, so each worktree gets a complete database in seconds. Use git-treeline release --drop-db to clean up when you're done.

Does it work with AI coding agents?

Yes. Git Treeline is designed for it. Add git-treeline setup . as a lifecycle hook when an agent creates a worktree, and git-treeline release . --drop-db on teardown. Use --json output for programmatic access to allocated resources.

What happens if I forget to clean up?

git-treeline prune --stale scans for allocations whose worktree directories no longer exist and removes them. It's garbage collection for your dev environment.
