Every worktree gets a URL

Git Treeline's networking layer gives each branch a named URL — locally, on your LAN, or publicly — without touching your app's code. No port numbers to remember. No config per branch.

gtl serve — Local HTTPS subdomain router

Maps {project}-{branch}.localhost to the correct worktree port. Runs as a launchd (macOS) or systemd (Linux) background service. Routes refresh every 5 seconds from the registry — new worktrees appear automatically.

Install (one-time)

The install command generates trusted HTTPS certificates, adds port forwarding for clean URLs (no port number), and registers the background service. Your system password is needed twice — once to trust the certificate authority, once to forward port 443.

$ gtl serve install

gtl serve is a local HTTPS router that gives every worktree a URL:

https://{project}-{branch}.localhost

It runs as a background service, routes traffic by subdomain,

and generates trusted HTTPS certificates automatically.

To set this up, your system password will be needed twice:

1. To trust a certificate authority so browsers accept *.localhost

2. To forward port 443 → 3001 so URLs don't need a port number

All set! Router is running in the background.

Navigating to bare https://localhost shows a clickable route table — every active worktree with a direct link to its URL.

How it works

  1. gtl serve install generates a local certificate authority and a wildcard *.localhost server certificate.
  2. The CA is trusted in your system keychain so browsers show a green lock.
  3. Port 443 is forwarded to the router (default port 3001) so URLs don't need a port number.
  4. A background service starts the router automatically on boot.
  5. The router reads the git-treeline registry every 5 seconds and updates its route table.
  6. Incoming requests are matched by subdomain and reverse-proxied to the correct local port. The Host header is preserved.

Uninstall

Remove everything cleanly:

$ gtl serve uninstall

Router service removed.

Port forwarding removed.

CA trust removed.

gtl proxy — Single-port reverse proxy

External services — OAuth providers, Stripe webhooks, CORS policies — often expect a fixed URL like localhost:3000. But Treeline allocates dynamic ports per worktree. gtl proxy bridges the gap: bind a fixed listen port and forward traffic to whichever worktree you're working in.

Usage

# Forward :3000 to the current worktree's allocated port

$ gtl proxy 3000

Proxying :3000 → :3010

# Explicit target port

$ gtl proxy 3000 3010

# With local HTTPS

$ gtl proxy 443 --tls

Proxying https://localhost:443 → :3010

Auto-detection

When you omit the target port, gtl proxy reads the registry for the current directory and forwards to the first allocated port. Run from inside a worktree that has been set up with gtl setup.

TLS certificates

The --tls flag terminates HTTPS on the listen side and forwards plain HTTP to the backend. Certificate resolution:

  1. Cached cert in ~/.config/git-treeline/certs/
  2. mkcert if installed — generates browser-trusted localhost certs
  3. Self-signed fallback (ECDSA P-256, 1 year, browsers will warn)

Protocol support

The proxy handles HTTP, WebSocket upgrades, and SSE streams transparently. The Host header from the original request is preserved, which matters for OAuth and virtual-host routing.

gtl tunnel — Public HTTPS via Cloudflare

Expose a running worktree to the public internet with a single command. Under the hood, Git Treeline manages cloudflared tunnels so you don't have to write config files or remember CLI flags.

Quick mode (no account needed)

Get a random public URL in seconds. No Cloudflare account, no setup. The URL is disposable — it changes every time you start the tunnel.

~/feature-auth $ gtl tunnel

Tunneling port 3010

https://random-slug-here.trycloudflare.com

Named mode (your domain)

After one-time setup, gtl tunnel automatically maps your branch to a subdomain on your own domain. The subdomain uses the same route key as gtl serve, so local and public URLs share the same naming convention.

~/feature-auth $ gtl tunnel

Tunneling port 3010 as myapp-feature-auth

https://myapp-feature-auth.myteam.dev

One Cloudflare tunnel handles all your projects. Each gtl tunnel invocation writes a per-run config that routes the correct subdomain to the correct local port.

How named tunnels work

  1. gtl tunnel setup creates a Cloudflare tunnel and a wildcard DNS record (*.myteam.dev) pointing at the tunnel.
  2. When you run gtl tunnel, it resolves your project name and branch, computes a route key, and writes a per-run cloudflared config file.
  3. The config routes {routeKey}.{domain} to http://localhost:{port}.
  4. Cloudflare terminates TLS at the edge. The local connection is plain HTTP.
  5. If project or branch can't be determined, it falls back to quick mode with a random URL.

Requirements

Quick mode needs cloudflared installed (no account). Named mode needs a Cloudflare account with a domain. On macOS, gtl tunnel setup offers to install cloudflared via Homebrew if it's missing.

Route key convention

Both gtl serve and gtl tunnel derive URLs from the same route key. The key is computed from your project name (from .treeline.yml) and the current git branch:

  1. Project and branch names are lowercased
  2. Common branch prefixes are stripped: feature/, chore/, bugfix/, hotfix/
  3. Slashes and underscores become dashes
  4. Non-alphanumeric characters (except dashes) are removed
  5. Consecutive dashes are collapsed
  6. Result: {project}-{branch}

Examples

Branch Route key Local URL Tunnel URL
main myapp-main myapp-main.localhost myapp-main.myteam.dev
feature/auth-flow myapp-auth-flow myapp-auth-flow.localhost myapp-auth-flow.myteam.dev
bugfix/fix_login_crash myapp-fix-login-crash myapp-fix-login-crash.localhost myapp-fix-login-crash.myteam.dev
chore/deps-2026-04 myapp-deps-2026-04 myapp-deps-2026-04.localhost myapp-deps-2026-04.myteam.dev

The route key is also shown in gtl setup output when the router or tunnel is configured.

Config reference

Networking settings live in user config (gtl config), not in .treeline.yml. They're per-machine, not per-project.

Setting Default Description
router.port 3001 Port the local router listens on (before port forwarding from 443). Kept off 3000 so gtl proxy can bind :3000 for OAuth/webhook use cases.
tunnel.name — Cloudflare tunnel name. Set by gtl tunnel setup. One tunnel handles all projects.
tunnel.domain — Domain for named tunnel subdomains (e.g. myteam.dev). Set by gtl tunnel setup.

CLI reference

Command Flags Description
serve Start the subdomain router in the foreground (for testing).
serve install One-time setup: generate certs, trust CA, forward port 443, install background service.
serve uninstall Remove service, CA trust, and port forwarding.
serve status Show router state, CA status, port forwarding, and active routes.
proxy <listen-port> [target] --tls Reverse proxy from listen port to target. Target defaults to current worktree's allocated port.
tunnel [port] --domain Expose a port via Cloudflare tunnel. Port defaults to current worktree. Named mode if domain configured.
tunnel setup Interactive: install cloudflared, authenticate, create tunnel, configure DNS, save config.
tunnel status Show cloudflared install, auth, tunnel, and domain status.
