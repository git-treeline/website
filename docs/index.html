Getting started

Git Treeline is a standalone CLI binary — no runtime dependency, no git hooks, no background processes. It writes an env file at setup time and gets out of the way.

New here? See how isolation works for the full picture.

Install

Homebrew (macOS and Linux):

$ brew install git-treeline/tap/git-treeline

Go install:

$ go install github.com/git-treeline/git-treeline@latest

Or download a binary from the releases page. The binary installs as git-treeline with a short alias gtl. It also works as git treeline.

Initialize a project

$ cd ~/myapp

$ gtl init --project myapp

Detected Rails project

Wrote .treeline.yml

Wrote AGENTS.md (treeline section)

gtl init auto-detects your framework (Rails, Next.js, Vite, Express, Python, Rust, Go) and generates a tailored .treeline.yml with commented-out optional config so available features are discoverable. It also writes a treeline section to AGENTS.md (or appends to CLAUDE.md) so Cursor, Claude Code, and Codex agents know to use gtl port. After generating the config, it prints actionable diagnostics — for example, Vite projects get vite.config.js port wiring guidance, Node projects without dotenv get a warning. Pass --skip-agent-config to skip the agent context file.

Create your first worktree

$ gtl new feature-auth

Creating worktree feature-auth

Allocated port 3010

Cloned database myapp_dev_feature_auth

Wrote .env.local

Running setup commands...

Ready. Start your server in this worktree.

Add --start to boot the app immediately after setup.

Review a GitHub PR

$ gtl review 42 --start

Fetching PR #42 (Add OAuth login)

Creating worktree myapp-pr-42

Allocated port 3020

Cloned database myapp_dev_pr_42

Starting server on port 3020...

Requires the GitHub CLI (gh). Both review and new must be run from the main repo, not from inside a worktree.

Switch branches in a worktree

$ gtl switch feature-payments

Fetching from origin...

Switched to feature-payments

Refreshed .env.local

Fetches from origin, checks out the branch, updates the registry, and refreshes the env file. Accepts PR numbers too — gtl switch 42 resolves the branch via gh. Add --setup to re-run commands.setup after switching.

Check project health

$ gtl doctor

Checks config presence, env file status, port allocation, supervisor state, and framework-specific guidance in one view. Run it when something seems off.

Configuration

.treeline.yml lives in your project root and is committed to the repo.

See Isolation for what each field produces and why it matters.

Full reference

Field Type Description
project string Project name. Used as a namespace for allocations and database naming.
merge_target string Branch that prune --merged checks against. Auto-detected from remote if omitted.
port_count int Number of ports to allocate per worktree. Default 1. Use 2 for apps with a separate asset build port.
env_file string or map Env file path as a string shorthand (e.g. .env.local). When the output file differs from the seed, use a map: path: (where values are written) and seed_from: (source template to copy first).
database.adapter string postgresql or sqlite. Determines cloning strategy.
database.template string Source database to clone from. For PostgreSQL, used with createdb --template.
database.pattern string Naming pattern for cloned databases. Tokens: {template}, {worktree}.
redis.strategy string prefixed (key namespacing, unlimited) or database (separate Redis DBs 1–15).
copy_files list Files to copy from the main worktree. Secrets, credentials, local config that aren't in git.
env map Key-value pairs written to the env file. Supports interpolation tokens.
commands.setup list Commands to run after env file is written. Allocated values like $PORT are available. See setup commands.
commands.start string Command to boot the app. Used by gtl start and the --start flag on new and review.
editor.title string Window title template. Tokens: {project}, {port}, {branch}. Works with VS Code and Cursor.
editor.color string Title/status bar color. "auto" generates a deterministic color from the branch name. Or set a hex value like "#1a5276". VS Code, Cursor, JetBrains.
editor.theme string Full IDE theme override (e.g. "Monokai"). Applied when entering the worktree.

Interpolation tokens

Use these tokens in the env map. They're replaced with allocated values at setup time.

Token Value
{port} Primary allocated port
{port_2}, {port_3}, ... Additional allocated ports (when port_count > 1)
{database} Cloned database name
{redis_url} Redis URL with namespace applied
{redis_prefix} Redis key prefix (when using prefix strategy)
{redis_db} Redis database number (when using database strategy)
{project} Project name from config
{worktree} Worktree directory name

CLI reference

Invoked as git-treeline, git treeline, or gtl. All three are equivalent.

init
--project--template-db--skip-agent-config

Generate .treeline.yml with self-documenting template. Auto-detects framework, writes AGENTS.md section, detects editor.

new <branch>
--base--path--start--dry-run

Create worktree + allocate resources + run setup in one step. --start boots the app after setup. Must be run from the main repo.

review <PR#>
--path--start

Fetch a GitHub PR into a new worktree with full setup. Requires gh CLI. Must be run from the main repo.

switch <branch-or-PR#>
--setup

Switch a worktree to a different branch or PR. Fetches from origin, checks out, updates registry, refreshes env. --setup re-runs commands.setup.

setup [path]
--main-repo--dry-run

Allocate resources and configure a worktree. Idempotent — safe to re-run.

refresh
--dry-run--force

Re-allocate all worktrees with current config and reservations. Restarts supervised servers automatically; flags manually-started servers.

release [path]
--drop-db--project--all--force--dry-run

Free allocated resources. --drop-db removes the cloned database. --all releases everything across projects.

port

Print the allocated port for the current worktree. Designed for agents and scripts.

doctor

Check config, allocation, runtime, and framework-specific diagnostics in one view.

status
--project--json--check--watch--interval

Show all allocations. --check probes ports to verify servers are responding. --watch auto-refreshes at --interval (default 2s). --json for structured output.

prune
--stale--merged--drop-db--force

Clean orphaned allocations. --stale targets worktrees whose directory no longer exists. --merged targets branches merged to merge_target. --drop-db also drops databases.

config list

Dump all user-level settings.

config get <key>

Read a setting by dot-notation key (e.g. port.base, redis.url).

config set <key> <value>

Write a setting and persist to disk.

config path

Print the config file location.

config edit

Open config in $EDITOR.

db name

Print the worktree's database name.

db reset
--from <db>

Drop and re-clone from template. --from overrides the source database.

db restore <file>

Drop and restore from a pg_dump file. Auto-detects custom format vs plain SQL.

db drop

Drop the worktree's database without re-cloning.

start

Run commands.start under a Unix socket supervisor. Injects allocated env vars into the child process. Resumes the server if the supervisor is already running.

stop

Pause the supervised server. The supervisor stays alive so the server can be resumed.

restart

Atomic stop + start. Bounces the supervised server in one step.

editor refresh

Re-apply editor title, color, and theme settings for the current worktree's branch.

serve

Local HTTPS subdomain router. serve install for one-time setup, serve status for routes. Details →

proxy <port> [target]
--tls

Single-port reverse proxy. Target defaults to current worktree's allocated port. Details →

tunnel [port]
--domain

Public HTTPS tunnel via Cloudflare. tunnel setup for one-time config. Details →

mcp

Start the MCP server (stdio transport). Used by editors, not invoked directly. Details →

version

Print version.

User config

User-level settings control allocation behavior across all projects on your machine. Manage them with gtl config list, gtl config get <key>, and gtl config set <key> <value>. Use gtl config path to find the file and gtl config edit to open it in your editor.

Platform Path
macOS ~/Library/Application Support/git-treeline/
Linux $XDG_CONFIG_HOME/git-treeline/ (defaults to ~/.config/git-treeline/)
Windows %APPDATA%/git-treeline/
Setting Default Description
port.base 3000 Starting port number for allocations.
port.increment 10 Gap between port allocations. Set to match your port_count or higher.
redis.strategy prefixed prefixed for key namespacing (unlimited worktrees) or database for separate Redis DBs (1–15).
redis.url redis://localhost:6379 Redis server URL used for namespace assignment.
port.reservations {} Map of project (or project/branch) to pinned port number. Branch-specific keys take priority.
editor.name (auto) Detected editor (Cursor, VS Code, Zed, JetBrains). Set by gtl init; used by the menulet.
editor.themes {} Per-project or per-branch theme overrides. Keys are project or project/branch.
editor.colors {} Per-project or per-branch color overrides. Same key format as themes.

Port reservations

Pin stable ports to specific projects or branches so they never change between allocations. A project-level key applies to the main repo. A project/branch key pins a specific worktree — useful for long-lived worktrees that need a known port.

$ gtl config set port.reservations.myapp 3000

$ gtl config set port.reservations.myapp/staging 3020

Reserved ports block the full port.increment range and are excluded from the dynamic pool. After adding reservations, run gtl refresh to re-allocate existing worktrees.

Framework integration

Git Treeline writes the env file your stack already loads. No extra dependencies. gtl init auto-detects your framework and generates the right config.

Rails

Rails apps using dotenv-rails (most do) load .env.local at boot automatically. Git Treeline writes allocated values to that file, so ENV["PORT"], ENV["DATABASE_NAME"], and ENV["REDIS_URL"] just work. No additional gems needed.

If your Rails app uses a JS bundler (jsbundling-rails, cssbundling-rails, or a multi-process Procfile.dev), gtl init detects it and sets port_count: 2 with an ESBUILD_PORT env var for the asset process.

Next.js

Next.js loads .env.local into process.env for your app code, but the dev server does not use PORT for its listen port. Update your package.json dev script: 

"dev": "next dev --port ${PORT:-3000}"

gtl init prints this guidance automatically for Next.js projects.

Vite (React, Vue, Svelte, etc.)

gtl init auto-detects Vite projects (vite.config.js, vite.config.ts, vite.config.mjs) and generates a tailored config with .env.local wiring and npx vite as the start command.

Vite loads .env.local for import.meta.env in your app code, but does not use PORT for the dev server port by default. Add this to your vite.config.js:

vite.config.js 
import { defineConfig, loadEnv } from 'vite'
export default defineConfig(({ mode }) => {
  const env = loadEnv(mode, process.cwd(), '')
  return {
    server: { port: parseInt(env.PORT || '5173') }
  }
})

gtl init prints this guidance automatically. Alternatively, if you use gtl start, the supervisor injects PORT directly into the process environment, so npx vite --port $PORT works without any vite.config.js changes.

Express / Node

Most Express apps already read process.env.PORT. Set env_file: .env and your app picks up the allocated port at startup. If your app seeds from .env.example, use the extended form: env_file: { path: .env, seed_from: .env.example }.

Other frameworks

Any framework that reads environment variables from an env file works with Git Treeline. Set env_file to the file your framework loads and define your env map with the tokens your app needs.

Setup commands

The commands.setup list runs after every resource allocation. Dependency installs, migrations, data fixups — anything your worktree needs before it's ready to boot.

Execution order

When gtl new or gtl setup runs, Git Treeline processes resources in this order:

  1. Allocate ports, database name, Redis namespace
  2. Copy files from main worktree (copy_files)
  3. Write env file with interpolated values
  4. Clone database (skipped on reused allocations)
  5. Run commands.setup with allocated values in the environment
  6. Configure editor

By the time your setup commands run, $PORT, $DATABASE_NAME, and every other key from your env map are available as environment variables. Setup commands always run — even on re-setup with a reused allocation.

Port-dependent data

Some apps store URLs that include the port — OAuth redirect URIs, webhook endpoints, CORS origins, API base URLs in database records. Those values go stale when Git Treeline allocates a new port. Use commands.setup to patch them.

Example: OAuth redirect URI (Doorkeeper)

A Rails app seeds an OAuth application with redirect_uri: "http://localhost:3000/callback". When Treeline allocates port 3010 for a worktree, that URI is wrong. Add a fixup:

.treeline.yml 
commands:
  setup:
    - bundle install --quiet
    - bin/rails db:migrate
    - bin/fix-port-data
bin/fix-port-data 
#!/usr/bin/env ruby
port = ENV.fetch("PORT")
app = OauthApplication.find_by(name: "Dev")
app&.update!(redirect_uri: "http://localhost:#{port}/callback")

The same pattern works for webhook URLs, CORS allowlists, asset host configs, or any record that embeds a port number. The script reads $PORT from the environment — already set by the time setup commands execute.

Re-setup behavior

Running gtl release followed by gtl setup on the same worktree produces a fresh allocation: new port, new env file, new database clone. Setup commands run again with the new $PORT in the environment, so the fixup script patches data to match.

On a reused allocation (e.g. running gtl setup without releasing first), Treeline skips the database clone but still runs setup commands. This makes setup safe to re-run — deps get installed, migrations run, and port-dependent data stays correct.

Database management

The gtl db commands manage worktree databases after initial setup. Reset to a clean state, restore from a dump, or clone from a different source.

Reset to template

Drop the worktree's database and re-clone from the configured database.template:

$ gtl db reset

Dropping myapp_dev_feature_auth

Cloning from myapp_development

Done.

Clone from a different source

Use --from to clone from a different database instead of the default template. Useful when a teammate has a clean database or you've pulled a sanitized staging dump:

$ gtl db reset --from staging_snapshot

Restore from a dump

Restore from a pg_dump file. Git Treeline auto-detects the format — custom format uses pg_restore, plain SQL uses psql:

# custom format (binary)

$ gtl db restore dump.pgcustom

# plain SQL

$ gtl db restore dump.sql

Other commands

db name

Print the worktree's allocated database name. Useful in scripts or to verify what's allocated.

db drop

Drop the database without re-cloning. Use when you want to free disk space without replacing the database.

Process supervisor

gtl start runs commands.start under a Unix socket supervisor. The supervisor injects allocated env vars directly into the child process. Logs stream to your terminal; other processes send stop/restart remotely.

See Workflows for the full story on PR review and the supervisor.

How it works

  1. You run gtl start in your terminal. The supervisor starts and launches your dev server.
  2. Server output streams to your terminal. You see every log line.
  3. An agent (or any process) calls gtl restart from another terminal. The supervisor bounces the server.
  4. Logs resume in your original terminal. No detached sessions, no hidden processes.
  5. Ctrl+C in your terminal fully exits the supervisor and the server.

Commands

Command Behavior
gtl start Launches supervisor + server. If supervisor is already running, resumes the server process.
gtl stop Stops the server process. Supervisor stays alive so the server can be resumed.
gtl restart Atomic stop + start. Bounces the server in one step.
Ctrl+C Fully exits the supervisor and the server. The socket is cleaned up.

Configuration

The supervisor uses the commands.start field from your .treeline.yml:

.treeline.yml 
commands:
  start: bin/dev

Security

The supervisor communicates over a Unix socket with 0600 permissions (owner-only). Read deadlines and client timeouts prevent hung connections. The socket path is derived from a SHA-256 hash of the worktree path.

Editor customization

Git Treeline can set window titles, title bar colors, and themes per worktree so you always know which branch you're looking at. Works with VS Code, Cursor, Zed, and JetBrains IDEs.

Project config

Add editor settings to your .treeline.yml:

.treeline.yml 
editor:
  title: "{project} ({branch}) :{port}"
  color: "auto"
  theme: "GitHub Dark"
Setting Behavior
title Window title template. Tokens: {project}, {port}, {branch}.
color "auto" generates a deterministic color from the branch name so each worktree looks different. Or set a hex value like "#1a5276".
theme Full IDE theme name (e.g. "Monokai"). Applied when the worktree is active.

IDE support

VS Code / Cursor

Writes to .vscode/settings.json. If a .code-workspace file references the worktree, settings are written there instead.

JetBrains

If .idea/ exists, editor.color sets the project header color in workspace.xml (JetBrains 2023.2+).

Per-branch overrides

Override themes and colors for specific projects or branches via user config:

$ gtl config set editor.themes.myapp "Monokai"

$ gtl config set editor.colors.myapp/staging "#7b241c"

User config overrides take priority over project config. Keys are project or project/branch.

AI agent setup

Lifecycle hooks, structured output, MCP server, and auto-generated context files.

See AI Agents for use cases and integration patterns.

Lifecycle hooks

Map these to your agent orchestrator's worktree creation and teardown events:

# on worktree creation

$ gtl setup .

# on worktree teardown

$ gtl release . --drop-db

Conductor / orchestrator config

If your orchestrator uses a JSON config for lifecycle hooks: 

{
  "setup": "gtl setup .",
  "archive": "gtl release . --drop-db"
}

Structured output

Use --json on any read command for machine-parseable output:

$ gtl status --json

[{
  "worktree": "agent-task-42",
  "port": 3010,
  "database": "myapp_dev_task_42",
  "redis_prefix": "myapp:task-42"
}]

Agent context file

gtl init writes a treeline section to AGENTS.md (created if it doesn't exist, appended otherwise). If CLAUDE.md exists but AGENTS.md doesn't, it appends there instead. The section tells agents to use gtl port for port discovery and lists the allocated env vars.

AGENTS.md is read by Cursor, Claude Code, and Codex — one file covers all three.

Port discovery

Agents can query the allocated port programmatically: 

PORT=$(gtl port)
curl http://localhost:$PORT/health

Use --skip-agent-config on gtl init to opt out of context file generation.

MCP server

Git Treeline ships a built-in Model Context Protocol server. Agents in Cursor, Claude Code, Windsurf, and other MCP-compatible editors get native tool access — no shell commands, no output parsing.

Add it to your editor's MCP configuration: 

{
  "mcpServers": {
    "gtl": {
      "command": "gtl",
      "args": ["mcp"]
    }
  }
}

Networking

Three tools give each worktree a named URL — locally, on your LAN, or publicly.

See Networking for the problem this solves and when to use each tool.

gtl serve

Local HTTPS subdomain router

Maps {project}-{branch}.localhost to the correct worktree port. Runs as a launchd (macOS) or systemd (Linux) background service. Routes refresh every 5 seconds from the registry.

Install (one-time)

Generates trusted HTTPS certificates, adds port forwarding for clean URLs, and registers the background service. Your system password is needed twice — once to trust the CA, once to forward port 443.

$ gtl serve install

gtl serve is a local HTTPS router that gives every worktree a URL:

https://{project}-{branch}.localhost

To set this up, your system password will be needed twice:

1. To trust a certificate authority so browsers accept *.localhost

2. To forward port 443 → 3001 so URLs don't need a port number

All set! Router is running in the background.

https://{project}-{branch}.localhost

Check status and routes

$ gtl serve status

Router: running on port 3001 (HTTPS)

CA: installed

Port forwarding: active (443 → router)

Routes (3):

https://myapp-main.localhost → :3000

https://myapp-feature-auth.localhost → :3010

https://myapp-pr-42.localhost → :3020

Navigating to bare https://localhost shows a clickable route table with direct links to every active worktree.

How it works

  1. gtl serve install generates a local CA and wildcard *.localhost server certificate.
  2. The CA is trusted in your system keychain so browsers show a green lock.
  3. Port 443 is forwarded to the router (default port 3001) so URLs don't need a port number.
  4. A background service starts the router automatically on boot.
  5. The router reads the git-treeline registry every 5 seconds and updates its route table.
  6. Incoming requests are matched by subdomain and reverse-proxied to the correct local port. The Host header is preserved.

Uninstall

$ gtl serve uninstall

Router service removed.

Port forwarding removed.

CA trust removed.

gtl proxy

Single-port reverse proxy

External services — OAuth providers, Stripe webhooks, CORS policies — often expect a fixed URL. gtl proxy binds a fixed listen port and forwards traffic to the current worktree's allocated port.

Usage

# Forward :3000 to the current worktree's allocated port

$ gtl proxy 3000

Proxying :3000 → :3010

# Explicit target port

$ gtl proxy 3000 3010

# With local HTTPS

$ gtl proxy 443 --tls

Proxying https://localhost:443 → :3010

Auto-detection

When you omit the target port, gtl proxy reads the registry for the current directory and forwards to the first allocated port. Run from inside a worktree that has been set up with gtl setup.

TLS certificates

The --tls flag terminates HTTPS on the listen side and forwards plain HTTP to the backend. Certificate resolution order:

  1. Cached cert in ~/.config/git-treeline/certs/
  2. mkcert if installed — generates browser-trusted localhost certs
  3. Self-signed fallback (ECDSA P-256, 1 year)

Protocol support

Handles HTTP, WebSocket upgrades, and SSE streams transparently. The Host header is preserved.

gtl tunnel

Public HTTPS via Cloudflare

Expose a running worktree to the public internet. Git Treeline manages cloudflared tunnels so you don't write config files or remember flags.

Quick mode (no account needed)

Disposable random URL in seconds. No Cloudflare account, no setup.

~/feature-auth $ gtl tunnel

Tunneling port 3010

https://random-slug-here.trycloudflare.com

Named mode (your domain)

After one-time setup, gtl tunnel automatically maps your branch to a subdomain on your domain. Uses the same route key as gtl serve.

~/feature-auth $ gtl tunnel

Tunneling port 3010 as myapp-feature-auth

https://myapp-feature-auth.example.dev

Setup walkthrough

gtl tunnel setup is a 5-step interactive wizard:

$ gtl tunnel setup

Step 1: Checking cloudflared

cloudflared found.

Step 2: Cloudflare authentication

Opening Cloudflare login in your browser...

Authenticated.

Step 3: Create tunnel

Tunnel name [gtl]: gtl

Tunnel created.

Step 4: Domain configuration

Domain (e.g. example.dev): example.dev

Routing *.example.dev → tunnel "gtl"...

DNS configured.

Step 5: Saving configuration

Saved tunnel.name=gtl, tunnel.domain=example.dev

Done! You can now run:

gtl tunnel

Example: https://myapp-feature-auth.example.dev

How named tunnels work

  1. gtl tunnel setup creates a Cloudflare tunnel and a wildcard DNS record (*.example.dev).
  2. When you run gtl tunnel, it resolves the project name and branch, computes a route key, and writes a per-run cloudflared config.
  3. The config routes {routeKey}.{domain} to http://localhost:{port}.
  4. Cloudflare terminates TLS at the edge. The local connection is plain HTTP.
  5. If project or branch can't be determined, it falls back to quick mode.

Requirements

Quick mode needs cloudflared installed (no account). Named mode needs a Cloudflare account with a domain. On macOS, gtl tunnel setup offers to install cloudflared via Homebrew if missing.

Route key convention

Both gtl serve and gtl tunnel derive URLs from the same route key. Computed from your project name (from .treeline.yml) and the current git branch:

  1. Project and branch names are lowercased
  2. Common branch prefixes stripped: feature/, chore/, bugfix/, hotfix/
  3. Slashes and underscores become dashes
  4. Non-alphanumeric characters (except dashes) removed
  5. Consecutive dashes collapsed
  6. Result: {project}-{branch}

Examples

Branch Route key URLs
main myapp-main myapp-main.localhost
myapp-main.example.dev
feature/auth-flow myapp-auth-flow myapp-auth-flow.localhost
myapp-auth-flow.example.dev
bugfix/fix_login_crash myapp-fix-login-crash myapp-fix-login-crash.localhost
myapp-fix-login-crash.example.dev

The route key is also shown in gtl setup output when the router or tunnel is configured.
