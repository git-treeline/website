Getting started

Git Treeline is a standalone CLI binary — no runtime dependency, no git hooks, no background processes. It writes an env file at setup time and gets out of the way. Team members who don't use it just ignore the config file.

Install

Homebrew (macOS and Linux):

$ brew install git-treeline/tap/git-treeline

Go install:

$ go install github.com/git-treeline/git-treeline@latest

Or download a binary from the releases page. The binary installs as git-treeline with a short alias gtl. It also works as git treeline.

Initialize a project

$ cd ~/myapp

$ gtl init --project myapp

Detected Rails project

Wrote .treeline.yml

Wrote .cursor/rules/treeline.mdc

gtl init auto-detects your framework (Rails, Next.js, Vite, Express, Python, Rust, Go) and generates a tailored .treeline.yml. After generating the config, it prints actionable diagnostics — for example, Vite projects get vite.config.js port wiring guidance, Node projects without dotenv get a warning. Pass --skip-agent-config to skip AI agent context files.

Create your first worktree

$ gtl new feature-auth

Creating worktree feature-auth

Allocated port 3010

Cloned database myapp_dev_feature_auth

Wrote .env.local

Running setup commands...

Ready. Start your server in this worktree.

Add --start to boot the app immediately after setup.

Review a GitHub PR

$ gtl review 42 --start

Requires the GitHub CLI (gh).

Configuration

.treeline.yml lives in your project root and is committed to the repo. Every team member and every agent gets the same setup.

Full reference

Field Type Description
project string Project name. Used as a namespace for allocations and database naming.
merge_target string Branch that prune --merged checks against. Auto-detected from remote if omitted.
ports_needed int Number of ports to allocate per worktree. Default 1. Use 2 for apps with a separate asset build port.
env_file.target string Output env file path. Where interpolated values are written.
env_file.source string Source env template to copy and merge with allocated values.
database.adapter string postgresql or sqlite. Determines cloning strategy.
database.template string Source database to clone from. For PostgreSQL, used with createdb --template.
database.pattern string Naming pattern for cloned databases. Tokens: {template}, {worktree}.
redis.strategy string prefixed (key namespacing, unlimited) or database (separate Redis DBs 1–15).
copy_files list Files to copy from the main worktree. Secrets, credentials, local config that aren't in git.
env map Key-value pairs written to the env file. Supports interpolation tokens.
commands.setup list Commands to run after env file is written. Allocated values like $PORT are available. See setup commands.
commands.start string Command to boot the app. Used by gtl start and the --start flag on new and review.
editor.vscode_title string VS Code window title template. Supports interpolation tokens.

Interpolation tokens

Use these tokens in the env map. They're replaced with allocated values at setup time.

Token Value
{port} Primary allocated port
{port_2}, {port_3}, ... Additional allocated ports (when ports_needed > 1)
{database} Cloned database name
{redis_url} Redis URL with namespace applied
{redis_prefix} Redis key prefix (when using prefix strategy)
{redis_db} Redis database number (when using database strategy)
{project} Project name from config
{worktree} Worktree directory name

CLI reference

Invoked as git-treeline, git treeline, or gtl. All three are equivalent.

Command Flags Description
init --project --template-db --skip-agent-config Generate .treeline.yml. Auto-detects framework and generates agent context files.
new <branch> --base --path --start --dry-run Create worktree + allocate resources + run setup in one step. --start boots the app after setup. --base sets the starting branch.
review <PR#> --path --start Fetch a GitHub PR into a new worktree with full setup. Requires gh CLI.
setup [path] --main-repo --dry-run Allocate resources and configure a worktree. Idempotent — safe to re-run.
refresh [path] Re-interpolate env file from existing allocation. No re-clone, no setup commands.
release [path] --drop-db --project --all --force --dry-run Free allocated resources. --drop-db removes the cloned database. --all releases everything across projects.
status --project --json --check --watch --interval Show all allocations. --project filters by project. --check probes ports. --watch auto-refreshes.
prune --stale --merged --drop-db --force Clean orphaned allocations. --stale targets missing directories. --merged targets branches merged to the merge target.
config list Dump all user-level settings.
config get <key> Read a setting by dot-notation key (e.g. port.base, redis.url).
config set <key> <value> Write a setting and persist to disk.
config path Print the config file location.
config edit Open config in $EDITOR.
db name Print the worktree's database name.
db reset --from <db> Drop and re-clone from template. --from overrides the source database.
db restore <file> Drop and restore from a pg_dump file. Auto-detects custom format vs plain SQL.
db drop Drop the worktree's database without re-cloning.
start Run commands.start under a Unix socket supervisor. Injects allocated env vars into the child process. If the supervisor is already running, resumes the server.
stop Pause the supervised server. The supervisor stays alive so the server can be resumed.
restart Atomic stop + start. Bounces the supervised server in one step.
version Print version.

User config

User-level settings control allocation behavior across all projects on your machine. Manage them with gtl config list, gtl config get <key>, and gtl config set <key> <value>. Use gtl config path to find the file and gtl config edit to open it in your editor.

Platform Path
macOS ~/Library/Application Support/git-treeline/
Linux $XDG_CONFIG_HOME/git-treeline/ (defaults to ~/.config/git-treeline/)
Windows %APPDATA%/git-treeline/
Setting Default Description
port.base 3000 Starting port number for allocations.
port.increment 10 Gap between port allocations. Set to match your ports_needed or higher.
redis.strategy prefixed prefixed for key namespacing (unlimited worktrees) or database for separate Redis DBs (1–15).
redis.url redis://localhost:6379 Redis server URL used for namespace assignment.

Framework integration

Git Treeline writes the env file your stack already loads. No extra dependencies. gtl init auto-detects your framework and generates the right config.

Rails

Rails apps using dotenv-rails (most do) load .env.local at boot automatically. Git Treeline writes allocated values to that file, so ENV["PORT"], ENV["DATABASE_NAME"], and ENV["REDIS_URL"] just work. No additional gems needed.

If your Rails app uses a JS bundler (jsbundling-rails, cssbundling-rails, or a multi-process Procfile.dev), gtl init detects it and sets ports_needed: 2 with an ESBUILD_PORT env var for the asset process.

Next.js

Next.js loads .env.local into process.env for your app code, but the dev server does not use PORT for its listen port. Update your package.json dev script: 

"dev": "next dev --port ${PORT:-3000}"

gtl init prints this guidance automatically for Next.js projects.

Vite (React, Vue, Svelte, etc.)

gtl init auto-detects Vite projects (vite.config.js, vite.config.ts, vite.config.mjs) and generates a tailored config with .env.local wiring and npx vite as the start command.

Vite loads .env.local for import.meta.env in your app code, but does not use PORT for the dev server port by default. Add this to your vite.config.js:

vite.config.js 
import { defineConfig, loadEnv } from 'vite'
export default defineConfig(({ mode }) => {
  const env = loadEnv(mode, process.cwd(), '')
  return {
    server: { port: parseInt(env.PORT || '5173') }
  }
})

gtl init prints this guidance automatically. Alternatively, if you use gtl start, the supervisor injects PORT directly into the process environment, so npx vite --port $PORT works without any vite.config.js changes.

Express / Node

Most Express apps already read process.env.PORT. Point env_file.target at .env and your app picks up the allocated port at startup.

Other frameworks

Any framework that reads environment variables from an env file works with Git Treeline. Set env_file.target to the file your framework loads and define your env map with the tokens your app needs.

Setup commands

The commands.setup list runs after every resource allocation. Dependency installs, migrations, data fixups — anything your worktree needs before it's ready to boot.

Execution order

When gtl new or gtl setup runs, Git Treeline processes resources in this order:

  1. Allocate ports, database name, Redis namespace
  2. Copy files from main worktree (copy_files)
  3. Write env file with interpolated values
  4. Clone database (skipped on reused allocations)
  5. Run commands.setup with allocated values in the environment
  6. Configure editor

By the time your setup commands run, $PORT, $DATABASE_NAME, and every other key from your env map are available as environment variables. Setup commands always run — even on re-setup with a reused allocation.

Port-dependent data

Some apps store URLs that include the port — OAuth redirect URIs, webhook endpoints, CORS origins, API base URLs in database records. Those values go stale when Git Treeline allocates a new port. Use commands.setup to patch them.

Example: OAuth redirect URI (Doorkeeper)

A Rails app seeds an OAuth application with redirect_uri: "http://localhost:3000/callback". When Treeline allocates port 3010 for a worktree, that URI is wrong. Add a fixup:

.treeline.yml 
commands:
  setup:
    - bundle install --quiet
    - bin/rails db:migrate
    - bin/fix-port-data
bin/fix-port-data 
#!/usr/bin/env ruby
port = ENV.fetch("PORT")
app = OauthApplication.find_by(name: "Dev")
app&.update!(redirect_uri: "http://localhost:#{port}/callback")

The same pattern works for webhook URLs, CORS allowlists, asset host configs, or any record that embeds a port number. The script reads $PORT from the environment — already set by the time setup commands execute.

Re-setup behavior

Running gtl release followed by gtl setup on the same worktree produces a fresh allocation: new port, new env file, new database clone. Setup commands run again with the new $PORT in the environment, so the fixup script patches data to match.

On a reused allocation (e.g. running gtl setup without releasing first), Treeline skips the database clone but still runs setup commands. This makes setup safe to re-run — deps get installed, migrations run, and port-dependent data stays correct.

Database management

The gtl db commands manage worktree databases after initial setup. Reset to a clean state, restore from a dump, or clone from a different source.

Reset to template

Drop the worktree's database and re-clone from the configured database.template:

$ gtl db reset

Dropping myapp_dev_feature_auth

Cloning from myapp_development

Done.

Clone from a different source

Use --from to clone from a different database instead of the default template. Useful when a teammate has a clean database or you've pulled a sanitized staging dump:

$ gtl db reset --from staging_snapshot

Restore from a dump

Restore from a pg_dump file. Git Treeline auto-detects the format — custom format uses pg_restore, plain SQL uses psql:

# custom format (binary)

$ gtl db restore dump.pgcustom

# plain SQL

$ gtl db restore dump.sql

Other commands

db name

Print the worktree's allocated database name. Useful in scripts or to verify what's allocated.

db drop

Drop the database without re-cloning. Use when you want to free disk space without replacing the database.

Process supervisor

gtl start runs the commands.start from your .treeline.yml under a Unix socket supervisor. The supervisor reads the worktree's allocation, resolves env templates, and injects them as real environment variables into the child process — PORT, DATABASE_URL, etc. are available without requiring the app to read .env files. Server logs stream to your terminal while other processes can send stop, start, and restart commands remotely.

How it works

  1. You run gtl start in your terminal. The supervisor starts and launches your dev server.
  2. Server output streams to your terminal. You see every log line.
  3. An agent (or any process) calls gtl restart from another terminal. The supervisor bounces the server.
  4. Logs resume in your original terminal. No detached sessions, no hidden processes.
  5. Ctrl+C in your terminal fully exits the supervisor and the server.

Commands

Command Behavior
gtl start Launches supervisor + server. If supervisor is already running, resumes the server process.
gtl stop Stops the server process. Supervisor stays alive so the server can be resumed.
gtl restart Atomic stop + start. Bounces the server in one step.
Ctrl+C Fully exits the supervisor and the server. The socket is cleaned up.

Configuration

The supervisor uses the commands.start field from your .treeline.yml:

.treeline.yml 
commands:
  start: bin/dev

Security

The supervisor communicates over a Unix socket with 0600 permissions (owner-only). Read deadlines and client timeouts prevent hung connections. The socket path is derived from a SHA-256 hash of the worktree path.

AI agent setup

Git Treeline integrates with AI coding agents that work in worktrees. Two hooks, structured output, and auto-generated context files.

Lifecycle hooks

Map these to your agent orchestrator's worktree creation and teardown events:

# on worktree creation

$ gtl setup .

# on worktree teardown

$ gtl release . --drop-db

Conductor / orchestrator config

If your orchestrator uses a JSON config for lifecycle hooks: 

{
  "setup": "gtl setup .",
  "archive": "gtl release . --drop-db"
}

Structured output

Use --json on any read command for machine-parseable output:

$ gtl status --json

[{
  "worktree": "agent-task-42",
  "port": 3010,
  "database": "myapp_dev_task_42",
  "redis_prefix": "myapp:task-42"
}]

Context file generation

gtl init generates context files that teach AI agents how to use Treeline in your project:

Cursor

Generates .cursor/rules/treeline.mdc with project-specific Treeline instructions.

Claude Code

Adds a Treeline section to CLAUDE.md with setup and teardown commands.

Use --skip-agent-config on gtl init to opt out.
